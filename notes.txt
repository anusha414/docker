Dockerfiles

1. For dotnet applications:-

For .Net apps, we usually use a multi-stage build:-
-> Build stage:- compiles the app(SDK image)
-> Runtime stage:- runs the app(lighter image)

example 1:-
# Stage 1: Build
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
# Copy project file and restore dependencies
COPY *.csproj .
RUN dotnet restore
# Copy everything else and build
COPY . .
RUN dotnet publish -c Release -o /app/publish
# Stage 2: Runtime
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
# Copy only published output
COPY --from=build /app/publish .
# Expose port
EXPOSE 80
# Run the app
ENTRYPOINT ["dotnet", "YourApp.dll"]

example 2:- Dockerfile for pre-built artifacts
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
# Copy published artifacts from pipeline
COPY publish/ .
EXPOSE 80
ENTRYPOINT ["dotnet", "YourApp.dll"]

Question 1:- What if there are multiple .csproj files in your repo? How will you identify the ENTRYPOINT DLL
Check the .csproj file if it contains "﻿<Project Sdk="Microsoft.NET.Sdk.Web">", it is runnable and is the entrypoint
if it contains "﻿<Project Sdk="Microsoft.NET.Sdk">", it is not runnable and is not the entrypoint
API/WEB means it is the ENTRYPOINT

Other than basic Dockerfile there are few important steps in dockerfile that must be taken care:-

# Base runtime image
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS base
# Set working directory
WORKDIR /app
# Create non-root user
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser
# Set environment variables
ENV ASPNETCORE_URLS=http://+:8080 \
   DOTNET_RUNNING_IN_CONTAINER=true \
   DOTNET_SYSTEM_GLOBALIZATION_INVARIANT=false
# Expose port
EXPOSE 8080
# Copy artifacts
COPY publish/ .
# Change ownership
RUN chown -R appuser:appgroup /app
# Switch to non-root user
USER appuser
# Health check (optional but important)
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
CMD curl --fail http://localhost:8080/health || exit 1
# Entry point
ENTRYPOINT ["dotnet", "YourApp.dll"]

for python application:-(Video:- https://www.youtube.com/watch?v=0UG2x2iWerk)

Prerequisites:-

Step 1:- findout which framework? flask, fastapi or django
Scan any python file for the imports like "from flask import Flask"
Usually only one framework will be used but there can be combinations of framework.

step 2 :- Findout the entrypoint
app.py---->app.run()-------->flask
main.py---->app=fastAPI()--->fastAPI
manage.py--->manage.py------>Django

NOTe:- app.py, main.py, manage.py are most commonly used but there is no contraints on python file naming conventions.

Docker file steps

step 1:- use the base image. https://hub.docker.com/_/python#image-variants
In our projects we are using slim images as base image because we are doing deployments for EKS deployments and production based deployments.

step 2:- identify dependencies, if u see a requiremnts.txt
COPY requirements.txt .
RUN pip install -r requirements.txt

step3:- based on the entrypoint create a CMD
Flask:- CMD [ "python", "app.py" ]
fastAPI:- CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
Django:- CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]

Step4:- find out ports
Flask:- look for app.run(port=5000)
FastAPI:- deafault(8000)
Django:- 8000

Examples:-

1. Flask
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
EXPOSE 5000
CMD ["python", "app.py"]

2. FastAPI
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

3. Django
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
