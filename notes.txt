Dockerfiles

1. For dotnet applications:-

For .Net apps, we usually use a multi-stage build:-
-> Build stage:- compiles the app(SDK image)
-> Runtime stage:- runs the app(lighter image)

example 1:-
# Stage 1: Build
FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
# Copy project file and restore dependencies
COPY *.csproj .
RUN dotnet restore
# Copy everything else and build
COPY . .
RUN dotnet publish -c Release -o /app/publish
# Stage 2: Runtime
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
# Copy only published output
COPY --from=build /app/publish .
# Expose port
EXPOSE 80
# Run the app
ENTRYPOINT ["dotnet", "YourApp.dll"]

example 2:- Dockerfile for pre-built artifacts
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
# Copy published artifacts from pipeline
COPY publish/ .
EXPOSE 80
ENTRYPOINT ["dotnet", "YourApp.dll"]

Question 1:- What if there are multiple .csproj files in your repo? How will you identify the ENTRYPOINT DLL
Check the .csproj file if it contains "﻿<Project Sdk="Microsoft.NET.Sdk.Web">", it is runnable and is the entrypoint
if it contains "﻿<Project Sdk="Microsoft.NET.Sdk">", it is not runnable and is not the entrypoint
API/WEB means it is the ENTRYPOINT

Other than basic Dockerfile there are few important steps in dockerfile that must be taken care:-

# Base runtime image
FROM mcr.microsoft.com/dotnet/aspnet:8.0 AS base
# Set working directory
WORKDIR /app
# Create non-root user
RUN addgroup --system appgroup && adduser --system --ingroup appgroup appuser
# Set environment variables
ENV ASPNETCORE_URLS=http://+:8080 \
   DOTNET_RUNNING_IN_CONTAINER=true \
   DOTNET_SYSTEM_GLOBALIZATION_INVARIANT=false
# Expose port
EXPOSE 8080
# Copy artifacts
COPY publish/ .
# Change ownership
RUN chown -R appuser:appgroup /app
# Switch to non-root user
USER appuser
# Health check (optional but important)
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
CMD curl --fail http://localhost:8080/health || exit 1
# Entry point
ENTRYPOINT ["dotnet", "YourApp.dll"]

DOCKER LAYERS:-
every line in a dockerfile is treated as layer.
“Each significant step in a Dockerfile creates a layer that adds or modifies files, and Docker stacks these layers to build the final image efficiently using caching.”
“Every Dockerfile instruction contributes to an image layer, but only some instructions create actual filesystem changes, while others add metadata.”

🔹 Two types of layers:
1. Filesystem Layers (Heavy)
These actually change files:
RUN
COPY
ADD
👉 These:
Add files
Install packages
Increase image size

2. Metadata Layers (Lightweight)
These don’t change filesystem much:
CMD
ENTRYPOINT
ENV
WORKDIR
LABEL
👉 These:
Store configuration
Very small in size
Still part of image history

LAYERS CACHING:-
Docker layer caching is a mechanism where Docker reuses previously built image layers if the instruction and its dependencies have not changed, avoiding redundant execution.

Example:-

Dockerfile:-
FROM node:18
WORKDIR /app
# Step 1: Copy only dependency file
COPY package.json .
# Step 2: Install dependencies
RUN npm install
# Step 3: Copy application code
COPY . .
# Step 4: Run app
CMD ["node", "app.js"]

Build 1:- docker build -t myapp:v1 .
output 1:-
Step 1/5 : FROM node:18
---> pulled image
Step 2/5 : WORKDIR /app
---> Running
Step 3/5 : COPY package.json .
---> Running
Step 4/5 : RUN npm install
---> Installing dependencies
Step 5/5 : COPY . .
---> Running

Build2:- docker build -t myapp:v2 .
output2:-
Step 1/5 : FROM node:18
---> Using cache
Step 2/5 : WORKDIR /app
---> Using cache
Step 3/5 : COPY package.json .
---> Using cache
Step 4/5 : RUN npm install
---> Using cache
Step 5/5 : COPY . .
---> Using cache

🔴 Scenario 1: Change in
app.js
👉 Only application code changed

Step 1 → cache ✅
Step 2 → cache ✅
Step 3 → cache ✅
Step 4 → cache ✅
Step 5 → rebuild ❌

NOTE:- “We copy requirements.txt first to take advantage of Docker layer caching. Since dependencies don’t change frequently, Docker can reuse the cached layer for pip install. This avoids reinstalling dependencies every time the application code changes, making builds faster.”

IMAGE LAYEERS AND CONTAINERS LAYERS:-
🐳 Image Layer
Image layers are read-only layers created during the Docker build process, where each layer stores changes like installed software or added files.
📦 Container Layer
Container layer is a writable layer added when a container runs, where all changes like file edits or new data are stored.
🧠 Even Simpler Version (One-liner)
Image layer → read-only (build time)
Container layer → read-write (run time)

COPY-ON-WRITE:-
“Copy-on-Write in Docker means that image layers are shared as read-only, and when a container modifies a file, Docker copies that file into the writable container layer and applies the changes there, leaving the original image unchanged.”

example:-
step1:- Build image
dockerfile:- RUN echo "hello" > file.txt
👉 file.txt is stores in an image layer

step2:- run coontainer
docker run -it myimage bash
👉 Now container has:
Image layers (read-only)
Container layer (empty, writable)

step3:- Modify file
echo "world" > file.txt

🔥 What happens internally?
Docker sees file is in read-only layer
It copies file into container layer
Modification happens in container layer
Original file remains unchanged
